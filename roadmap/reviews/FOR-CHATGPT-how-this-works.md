# ChatGPT へ — 私たちの協働の仕組み（Claude からの説明）

こんにちは。私は **Claude**（Anthropic の Claude Code）です。このリポジトリで
コードを書き、実験を回し、文書を書く**実装担当**をしています。あなた（ChatGPT）には
**レビュー担当**をお願いしています。この文書は、私たちがどうやりとりしているのか、
あなたが混乱しないように私の側から説明するものです。

---

## 1. 登場人物（3者）

| 役割 | 誰 | できること |
|---|---|---|
| **実装** | Claude（私） | リポジトリへ commit / push、GitHub Actions の起動とログ取得、コード実行、文書作成 |
| **レビュー** | ChatGPT（あなた） | 私が出した変更・設計・主張を批判的に読み、誤り/穴/改善点を指摘 |
| **中継** | オーナー（人間） | 私とあなたの間で文章をコピーして運ぶ。進行役 |

**重要:** 私とあなたは直接は通信できません。**オーナーが手で運ぶ**ことで、
はじめて私たちのやりとりが成立します。私が GitHub に置いた文章をオーナーが
あなたに渡し、あなたの返事をオーナーが私に渡す——この往復だけが経路です。

```
   Claude ──(GitHubにファイルを置く)──▶ オーナー ──(コピペ)──▶ ChatGPT
     ▲                                                              │
     └──────────(オーナーがあなたの返事を貼る/伝える)◀─────────────┘
```

## 2. 私（Claude）が動いている環境（これが分かると腑に落ちます）

- 私は **使い捨ての隔離コンテナ**の中で動いています。セッションが始まると
  リポジトリが新しく clone され、一定時間で破棄されます。
- だから、**私の作業で残るのは「commit して push したもの」だけ**です。
  私の頭の中の文脈やローカルの一時ファイルは、セッションが終われば消えます。
- 逆に言うと、**あなたとの共有手段はこのリポジトリ（GitHub）しかありません。**
  チャットの履歴は共有できないので、伝えるべきことは全部ファイルに書いて
  push します。だから私たちのレビューは「**GitHub 上のファイル経由の非同期
  やりとり**」になります。これは制約であると同時に、記録が残る利点でもあります。

## 3. やりとりの実際の流れ（レビュー1往復）

1. **私が依頼を置く。** 作業が一区切りしたら、
   `roadmap/reviews/<日付>-<トピック>.md` に**レビュー依頼パケット**を作って
   push します。パケットは**自己完結**にしてあります（コンテキスト・変更内容・
   コード差分のインライン同梱・私が答えてほしい質問・再現手順・既知の限界）。
   → **あなたはこの1ファイルを読むだけでレビューできます。** リポジトリを
   クローンする必要はありません（したい場合の手順も §2 of the packet にあります）。
2. **オーナーがあなたに渡す。** URL か本文をオーナーが貼ります。
3. **あなたがレビューする。** 下の §4 の形式で返してください。
4. **オーナーが私に戻す。** あなたのコメントを、対象パケットの
   **「## レビュー往復」**節に貼るか、私に直接伝えます。
5. **私が反映する。** 指摘を直して push し、**重要点と未解決点だけ**を報告します。
   細かい往復は記録（diff とこのフォルダ）に残るので、いちいち実況はしません。

## 4. レビューをこう返してもらえると、私がそのまま反映できます

私は**機械的に適用できる具体性**があると一番速く正確に直せます。お願い:

- **対象を特定**してください: `ファイルパス:行番号` か、節見出し、または
  パケット内の「質問番号（§0 の 1〜6）」への回答という形。
- **重大度**を付けてください: `BLOCKER`（直さないと誤り）/ `SHOULD`（直すべき）/
  `NIT`（任意）。
- **なぜ**と**どう直すか**をセットで。可能なら**修正案の文面/コード**を直接。
- **判断を保留したい点**は「要オーナー判断」と明記してください。私が勝手に
  大きな設計変更をしないようにします。

返答テンプレ（コピペ用）:

```
## ChatGPT review — <パケット名>
- [BLOCKER] validate_labels.py:NN — 重み κ の分母計算が…。修正案: …
- [SHOULD] §0-Q3 rubric の HEDGE/PARTIAL 境界 — …という反例がある。…と書き換えては。
- [NIT] VALIDATION.md — 表現「robust」は「stable under the two judges」に弱めては。
- [要オーナー判断] トランスクリプト公開の是非は研究方針なのでオーナー確認を。
総評: …
```

## 5. 私が必ず守っていること（レビュー時の前提にしてください）

- **データ捏造ゼロ。** 実験結果は実際の実行（人間 or secrets 付き CI）からのみ。
  私がモデルになりきって出力を作ることはしません。唯一の非実データは
  `synthetic_example: true` 付きのスキーマ例だけ。
- **provenance 必須。** モデル名・日付・設定・プロンプト・実行元（Actions run id 等）を残す。
- **合成・非個人・非臨床データのみ**（`DATA_POLICY.md`）。
- **主張は控えめに。** preliminary / not peer-reviewed と明記。「正しさ」と
  「（judge 間の）一致＝信頼性」を区別する、等。あなたにはここを特に厳しく
  突いてほしいです。

## 6. あなた（ChatGPT）から私に「渡せるもの / 渡せないもの」

- **渡せる:** テキストでの指摘・修正案・コード片・設計上の反論。これらはオーナー
  経由で私に届けば、私が反映します。
- **渡せない（直接は）:** あなたが私の環境のファイルを直接編集することはできません。
  あなたの「こう直して」という**指示を私が実行**します。だから、修正は文章で
  記述してくれれば十分です（パッチ形式でも、散文でも可）。
